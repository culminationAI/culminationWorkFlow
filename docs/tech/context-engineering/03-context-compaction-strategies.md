# Context Compaction: Стратегии сжатия

Как только сессия достигает лимита токенов, "Ultimate" система должна применить магию сжатия. Google выделяет несколько ключевых стратегий борьбы с бесконечным ростом данных.

## 1. Summarization (Суммаризация)
Это превращение длинного диалога в короткое резюме.
- **Incremental Summarization**: Каждые N сообщений мы обновляем краткий пересказ предыдущей истории.
- **Hierarchical Summarization**: Резюме резюме. Используется для сверхдлинных сессий (сотни сообщений).
- **Pros**: Сохраняет общий смысл.
- **Cons**: Мелкие, но важные детали могут быть потеряны.

## 2. Pruning (Избирательная обрезка)
Удаление наименее ценной информации.
- **Tool Trace Removal**: После того как агент получил результат поиска, сами шаги поиска (технические логи) можно удалить.
- **Few-shot Removal**: После первых нескольких обменов сообщениями модель уже "поняла" стиль, и примеры можно убрать для освобождения места.
- **Stopwords & Metadata Stripping**: Очистка технических заголовков.

## 3. Selective Context (Избирательный контекст)
Вместо того чтобы давать историю по порядку, мы выбираем только то, что относится к текущему вопросу.
- **Semantic Retrieval from Session**: Если пользователь спрашивает про «планы на вечер», мы достаем из сессии только фрагменты про время и планы, игнорируя пять страниц обсуждения погоды.

## 4. Трейд-оффы и Оптимизация
Сжатие — это баланс между тремя факторами:
1.  **Coherence (Связность)**: Чтобы агент не забыл, о чем шла речь 5 минут назад.
2.  **Latency (Задержка)**: Суммаризация требует дополнительных ресурсов LLM.
3.  **Cost (Стоимость)**: Лишние токены на обработку резюме.

## 5. Точки принятия решения (Triggers)
Когда запускать сжатие?
- **Token Threshold**: Достижение 70-80% окна контекста.
- **Event-based**: После завершения важной логической части (сцены, квеста).
- **Time-based**: При возобновлении старой сессии (например, на следующий день).

## 6. Стратегия компактизации для «Интерпретации»

### Что суммаризовать, что сохранять полностью

В проекте данные делятся на три категории по допустимости сжатия:

| Категория | Примеры | Стратегия | Причина |
|-----------|---------|-----------|---------|
| **Никогда не сжимать** | Речевой_профиль_героя, Ценности (top-5), активные Страхи, Роль | Постоянный system prompt | Определяют идентичность. Потеря = герой перестаёт быть собой |
| **Суммаризовать** | Цепочки Действие <-> Голос (диалоги), описания Сеттинга, логи Верификатора | Incremental summary после фазы | Детали важны в моменте, но не через 3 сцены |
| **Удалять** | Tool traces (Cypher-запросы к Neo4j), промежуточные расчёты Мониторинга, технические ответы Верификатора | Pruning после использования | Технический мусор. Результат сохранён в узлах графа |

### Пример: компактизация фазы «Встреча»

```python
# Псевдокод: компактизация после завершения фазы Encounter
def compact_encounter_phase(session, encounter_events):
    """
    Фаза Встреча генерирует ~3000 токенов (Отношение + итерации).
    После компактизации: ~400 токенов.
    """
    # 1. УДАЛИТЬ: технические трассы
    pruned = [e for e in encounter_events
              if e["type"] not in ("verification", "tool_trace", "graph_query")]

    # 2. СОХРАНИТЬ ПОЛНОСТЬЮ: активированные воспоминания
    #    (они влияют на всю оставшуюся сцену)
    preserved_memories = [e for e in pruned
                          if e["type"] == "activated_memory"]

    # 3. СУММАРИЗОВАТЬ: остальное
    summarizable = [e for e in pruned
                    if e["type"] not in ("activated_memory",)]

    summary = llm_summarize(
        text=summarizable,
        instruction="""Сожми фазу Встречи в 3-5 предложений.
        ОБЯЗАТЕЛЬНО сохрани:
        - Доминирующее направление внимания (внутрь/сеттинг/население/время)
        - Эмоциональный сдвиг героя (было -> стало)
        - Ключевое незамеченное (Незамеченное)
        - Отношение к каждому NPC (кратко)
        НЕ включай: описания локации (уже в узле Сеттинг),
        технические веса, промежуточные расчёты."""
    )

    return {
        "encounter_summary": summary,           # ~300 токенов
        "preserved_memories": preserved_memories, # ~100 токенов
        "tokens_saved": calculate_savings(encounter_events, summary),
    }
```

### Hierarchical Summarization: дневные сводки

В проекте Летописец формирует `Дневная_сводка` в конце каждого игрового дня. Это второй уровень суммаризации:

```
Уровень 0: Сырые события сцены          (~8000 токенов/сцена)
Уровень 1: Beat sheet (Летописец)        (~800 токенов/сцена)
Уровень 2: Дневная сводка (Летописец)    (~500 токенов/день, 3-5 сцен)
Уровень 3: Недельная арка (Драматург)    (~200 токенов/неделя)
```

Каждый уровень теряет детали, но сохраняет **эмоциональную кривую** и **ключевые дельты** (изменения весов в графе). Это прямая аналогия с тем, как работает человеческая память: мы не помним каждую минуту вторника, но помним, что «вторник был тяжёлым» и «на работе поругался с начальником».

### Триггеры компактизации в проекте

```python
COMPACTION_TRIGGERS = {
    # Токеновый порог: 70% от контекстного окна агента
    "token_threshold": lambda session: session.total_tokens > 0.7 * MODEL_CONTEXT_WINDOW,

    # Фазовый: между фазами пайплайна (естественные точки разрыва)
    "phase_boundary": lambda session, phase: phase.is_completed,

    # Событийный: после завершения цепочки Действие <-> Голос
    "action_chain_end": lambda chain: chain.depth == 0 and chain.is_settled,

    # Временной: при переходе к новой сцене (Хронос инициирует)
    "scene_transition": lambda transition: True,  # всегда компактим между сценами
}
```

### Психологическая параллель: консолидация памяти во сне

Стратегия «суммаризация между сценами» повторяет механизм консолидации памяти в фазе медленного сна. Гиппокамп «проигрывает» события дня, отбирая существенное для переноса в неокортекс (долговременную память). Несущественное — отбрасывается. В проекте роль гиппокампа выполняет Летописец: он «переигрывает» сцену через beat sheet, а затем через дневную сводку — уплотняя опыт до его смысловой сути.
