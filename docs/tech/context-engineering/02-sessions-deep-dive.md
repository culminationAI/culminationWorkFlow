# Sessions: Deep Dive в рабочую память

Сессия — это атомарная единица взаимодействия. Это «сейчас» в сознании агента. В документации Google сессии рассматриваются как контейнеры, обеспечивающие временную связность.

## 1. Анатомия Сессии
Сессия — это не просто текст. Это структурированный поток данных, включающий:

### A. Хронологический лог (Events)
Все взаимодействия в порядке их появления:
- **User Input**: Сырой запрос.
- **Agent Response**: Ответ модели.
- **Internal Monologue**: Мысли агента (если используются).
- **Tool Traces**: Вызовы функций и их результаты.

### B. Рабочий черновик (Scratchpad/State)
Область, где агент хранит временные переменные.
*Пример: Если агент выполняет расчет или ищет что-то в БД, он сохраняет результаты в Scratchpad, чтобы использовать их в нескольких шагах рассуждения (Chain of Thought).*

## 2. Разнообразие в фреймворках
Сессии могут отличаться в зависимости от архитектуры:
- **Message-based**: Простой список сообщений (OpenAI/Anthropic стиль).
- **State-based**: Графовое или табличное представление состояния (LangGraph/AutoGPT).

## 3. Мультиагентные сессии (Multi-Agent)
Когда в системе работают несколько агентов, управление сессиями усложняется:
- **Shared History**: Все агенты видят всё. (Ведет к быстрому Context Rot).
- **Individual Threads**: Каждый агент помнит только свои действия и то, что ему передали.
- **Broadcast/Handover**: Передача контекста от одного агента другому.

## 4. Интероперабельность
В "Ultimate" системах сессии должны быть переносимыми. 
- Пользователь может начать диалог в Telegram (один фреймворк) и продолжить в браузере (другой).
- Сессия должна храниться в универсальном формате (JSON/Protobuf), независимом от конкретного API поставщика LLM.

## 5. Продакшн-аспекты
Для масштабирования сессий Google рекомендует:
- **Fast-access storage**: Использование Redis для горячих сессий.
- **Session IDs**: Привязка к User ID, Device ID и Thread ID.
- **Expiration Policies**: Автоматическая очистка или архивация неактивных сессий (например, через 24 часа).

## 6. Анатомия сессии одной сцены в «Интерпретации»

Сцена в проекте — это не один вызов LLM. Это **оркестрированная последовательность из ~18-22 вызовов**, каждый из которых порождает события, трассы инструментов и изменения состояния. Рассмотрим конкретную сцену.

### Ключевые наблюдения

1. **Каждое событие типизировано.** Не просто «текст агента», а структурированный payload с конкретными полями. Это позволяет Selective Context: при обратном вызове Мониторинга мы берём только `emotional` и `physical`, не весь лог.

2. **Scratchpad — мозг сцены.** Веса внимания (`attention_weights`) определяют, куда пойдёт фаза «Встреча». Здесь `inward = 0.45` — значит, внутренний мир доминирует, будет рефлексия.

3. **Метаданные сессии** отслеживают рост контекста и готовят триггер компактизации. При `total_tokens > compaction_threshold` запускается суммаризация ранних фаз.

### Мультиагентная архитектура сессий в проекте

В «Интерпретации» используется паттерн **Individual Threads with Broadcast**:

```
Директор (координатор)
  |
  +-- Хронос:     собственный тред (время, темп)
  +-- Внутреннее: собственный тред (личность, память)
  +-- Мониторинг: собственный тред (состояния)
  +-- Сеттинг:    собственный тред (мир)
  +-- Население:  собственный тред (NPC)
  |
  +-- Отношение:  получает BROADCAST от Внутреннего + Мониторинга + Сеттинга + Населения
  +-- Действие:   получает BROADCAST от Отношения, может CALLBACK к любому
  +-- Голос:      получает BROADCAST от Действия, может CALLBACK к Мониторингу
  |
  +-- Верификатор: получает BROADCAST от всех (но проверяет, не запоминает)
```

Каждый агент видит только свою историю и то, что ему передали через broadcast. Это **архитектурная защита от Context Rot**: Голос никогда не видит полный лог Населения, только то, что прошло через фильтр Отношения.
