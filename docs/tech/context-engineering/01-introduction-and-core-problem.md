# Введение и основная проблема: Stateless LLMs

LLM по своей природе **ацикличны и лишены состояния (stateless)**. Каждый вызов API — это «чистый лист». Context Engineering — это дисциплина, которая решает эту фундаментальную проблему, превращая реактивную модель в самообучающегося агента.

## 1. Проблема: "Context Rot" (Тление контекста)
Без грамотного управления контекстом любая длинная сессия неизбежно деградирует.
- **Симптомы**: Агент начинает забывать инструкции, путает имена, игнорирует правила мира или повторяет одни и те же фразы.
- **Техническая причина**: Внимание модели (Attention) размывается по огромному массиву данных. Чем больше токенов в окне, тем слабее сигнал от каждого конкретного токена.

## 2. Экономика и Латентность
Просто хранить всю историю — дорого и медленно.
- **Bloated Prompts**: С каждым сообщением контекст растет. Если не управлять им, каждый ответ будет стоить дороже предыдущего.
- **Inference Delay**: Большой контекст требует больше вычислительных мощностей и времени на генерацию первого токена (TTFT).

## 3. От Prompt Engineering к Context Engineering
- **Prompt Engineering**: Фокусируется на *форме* запроса (как спросить).
- **Context Engineering**: Фокусируется на *содержимом* и *состоянии* (что именно дать модели в данный момент). 

### Состав типичного контекста (Payload):
1.  **System Instructions**: Правила игры и личность.
2.  **Tool Definitions**: Список доступных функций.
3.  **Few-shot Examples**: Примеры идеальных ответов.
4.  **Factual Data**: Данные из внешних источников (RAG).
5.  **Conversational History**: Сессия и память.

## 4. Цель "Ultimate" системы
Создать систему, где агент:
- **Помнит**: Сохраняет факты и эмоции.
- **Учится**: Меняет поведение на основе опыта.
- **Персонализирует**: Адаптируется под конкретного пользователя.

## 5. Context Rot в мультиагентной системе: проект «Интерпретация»

### Проблема: 12 агентов, 18-22 вызовов, растущий контекст

В «Интерпретации» одна сцена проходит через 7 фаз пайплайна (Контекст -> Внутреннее -> Действие -> Генерация -> Оценка -> Обновление -> Переход), задействуя до 12 агентов за ~18-22 LLM-вызовов. Каждый агент добавляет свой выход в общий контекст. Без управления контекст деградирует уже к фазе «Встреча».

**Анатомия деградации по фазам:**

```
Фаза 1 — Открытие сцены (~3 вызова):
  Хронос: +200 токенов (время, дата, сезон)
  Судьба:  +400 токенов (транзиты, модификаторы)
  Драматург: +500 токенов (фокус, значимость, чеховские ружья)
  Верификатор: +150 токенов (проверка)
  Итого: ~1 250 токенов
  Контекст: чистый, все данные релевантны

Фаза 2 — Загрузка героя (~3 вызова):
  Внутреннее: +800 токенов (3-5 ценностей, страхи, планы, воспоминания)
  Мониторинг: +600 токенов (38 эмоций, физика, когнитивные, роль)
  Верификатор: +150 токенов
  Нарастающий итого: ~2 800 токенов
  Контекст: ещё управляемый

Фаза 3 — Загрузка мира (~3 вызова):
  Сеттинг: +500 токенов (локация, объекты, атмосфера)
  Население: +700 токенов (NPC, профили, предметы)
  Верификатор: +150 токенов
  Нарастающий итого: ~4 150 токенов
  Контекст: начинает размываться — ранние инструкции Судьбы
  теряют вес в Attention

Фаза 4-5 — Встреча + Событие (~12 вызовов):
  Цепочка: Отношение -> Действие <-> Голос <-> Мониторинг (петли)
  Каждая итерация: +300-600 токенов
  Нарастающий итого: ~8 000-12 000 токенов
  === ЗОНА CONTEXT ROT ===
  Симптомы:
  - Герой "забывает" модификаторы Судьбы из фазы 1
  - Речь теряет характеристики из Речевого_профиля
  - Эмоциональные веса из Мониторинга игнорируются
  - NPC начинают вести себя вне своих профилей

Фаза 6 — Закрытие (~5 вызовов):
  Нарастающий итого: ~14 000-18 000 токенов
  Летописец работает с деградированным контекстом
```

### Решение: послойная изоляция контекста

В «Интерпретации» контекст не передаётся монолитно. Каждый агент получает только то, что ему нужно:

```python
# Псевдокод: формирование контекста для агента Голос (Agent 7)
def build_voice_context(scene, hero, current_action):
    """Голос получает ~2000 токенов, а не 14000."""
    return {
        # Из Внутреннего — только речевой профиль (постоянный)
        "speech_profile": graph.get_node("Речевой_профиль_героя"),

        # Из Мониторинга — только текущие речевые параметры
        "speech_state": {
            "вербальная_уверенность": monitor.get("Вербальная_уверенность"),
            "фильтр_речи": monitor.get("Фильтр_речи"),
            "красноречивость": monitor.get("Красноречивость"),
            "опьянение": monitor.get("Опьянение"),
        },

        # Из Действия — что именно порождает речь
        "action_trigger": current_action.summary,

        # Из Отношения — к кому обращена речь
        "addressee": current_action.target_npc.brief_profile,

        # Сожаления — чтобы не повторять ошибки
        "recent_regrets": monitor.get_active("Сожаление_о_фразе", limit=2),

        # НЕ включаем: полный список NPC, транзиты Судьбы,
        # всю эмоциональную панель, историю сцены
    }
```

### Психологическая параллель

Context Rot в LLM — это точный аналог **перегрузки рабочей памяти** в когнитивной психологии. Исследования Миллера (1956) показывают, что рабочая память удерживает 7 +/- 2 объектов. При перегрузке:
- Ранние элементы вытесняются (эффект новизны доминирует)
- Качество обработки каждого элемента падает
- Появляются «ложные воспоминания» — аналог галлюцинаций LLM

В «Интерпретации» мы проектируем агентов по тому же принципу, что и человеческое внимание: **в фокусе — 3-5 самых весомых ценностей, 2-3 активных страха** (см. агент Внутреннее). Не весь граф. Это не ограничение — это архитектурное решение, повторяющее работу реальной психики.
