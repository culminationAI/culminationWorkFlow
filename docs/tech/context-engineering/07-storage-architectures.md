# Архитектуры хранения: Графы vs Векторы

Выбор места хранения памяти определяет, как агент будет «думать». Google рассматривает три основные архитектуры, которые мы реализуем в нашем проекте.

## 1. Векторные БД (Semantic Memory)
*Пример: Qdrant*
- **Как работает**: Текст превращается в числа (эмбеддинги). Поиск идет по «смыслу», а не по словам.
- **Сильные стороны**: Идеально для «похожих воспоминаний», поиска по настроению и неструктурированных данных.
- **Слабые стороны**: Плохо справляется с точными фактами и сложными связями (например, "Кто брат жены Ивана?").

## 2. Графы Знаний (Factual Memory)
*Пример: Neo4j*
- **Как работает**: Данные хранятся как Узлы (Сущности) и Связи между ними.
- **Сильные стороны**: Точная логика, навигация по цепочкам отношений, 100% достоверность фактов.
- **Слабые стороны**: Требует жесткой схемы (DDL) или очень умного экстрактора.

## 3. Гибридная архитектура (The Winning Pattern)
Это то, что мы строим в «Интерпретации». 
1.  **Graph-based Context Retrieval**: Сначала мы идем в Neo4j, чтобы достать жесткие факты о герое.
2.  **Vector-based Semantic Search**: Затем мы идем в Qdrant, чтобы найти похожие эмоциональные моменты.
3.  **Synthesis**: LLM объединяет оба потока в один связный контекст.

## 4. Multi-modal Memory
Документация Google также упоминает память для изображений и звуков. 
- Хотя наш проект текстоцентричен, архитектура должна позволять хранить ссылки на `image_vectors` в том же Qdrant, чтобы агент мог «вспомнить» лицо персонажа или атмосферу места.
