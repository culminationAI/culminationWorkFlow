# 12. Протокол оптимизации запросов (Query Optimization Protocol)

> [!TIP]
> Верифицировано 2026-02-26. Основа: tier-система маршрутизации, встроенная в `CLAUDE.md`.

---

## Зачем

При подписке Max x20 основные потребители бюджета:
1. **Extended thinking** — до 200K output-токенов на один запрос
2. **Opus для каждого субагента** — даже для простых задач
3. **Отсутствие предварительной оценки** — каждый запрос обрабатывается на максимуме

Протокол устраняет все три проблемы.

---

## Tier-система классификации

Каждый запрос пользователя классифицируется по 5-балльной шкале:

```
T1 (Trivial)  → Sonnet, без thinking  → «Найди файл X»
T2 (Simple)   → Sonnet, effort: low   → «Добавь раздел Y»
T3 (Medium)   → Sonnet, effort: med   → «Напиши документ Z»
T4 (Complex)  → Opus,   effort: high  → «Спроектируй архитектуру W»
T5 (Expert)   → Opus,   effort: max   → «Исследуй тему V с анализом»
```

### Критерии классификации

| Критерий | T1 | T2 | T3 | T4 | T5 |
|----------|----|----|----|----|-----|
| Количество шагов | 1 | 2-3 | 4-8 | 8-20 | 20+ |
| Нужен ли анализ | Нет | Минимальный | Да | Глубокий | Исследовательский |
| Размер выхода | <500 tok | <2K tok | 2-5K tok | 5-15K tok | 15K+ tok |
| Субагенты | 0 | 0 | 0-1 | 1-3 | 3+ |
| Источники | 0 | 0 | 0-1 | 1-3 | 5+ |

## Маршрутизация моделей

### Главный агент (Claude Code)

| Tier | Thinking | Обоснование |
|------|----------|-------------|
| T1 | Выключен | Нет рассуждения → нет output-токенов на thinking |
| T2 | Выключен | Простые правки не требуют рассуждения |
| T3 | Low effort | Минимальное рассуждение для планирования |
| T4 | Medium effort | Глубокое рассуждение нужно, но не максимальное |
| T5 | High effort | Полная мощность для исследований |

### Субагенты

| Tier | Модель | Когда вызывать |
|------|--------|----------------|
| T1 | — | Субагенты НЕ нужны |
| T2 | — | Субагенты НЕ нужны |
| T3 | Sonnet | Один-два субагента, если задача в их домене |
| T4 | Opus | Субагенты с thinking, можно параллельно |
| T5 | Opus | Максимальная мощность, полная цепочка |

> [!IMPORTANT]
> Дефолт всех субагентов — `model: sonnet`. При tier 4-5 оркестратор эскалирует модель при вызове.

## Правила оптимизации запроса

### До выполнения — мысленная предобработка

1. **Извлечь ядро задачи** — что именно нужно сделать (одно предложение)
2. **Определить границы** — что НЕ нужно (не расширять скоуп)
3. **Установить формат** — длина, структура, язык ответа
4. **Оценить зависимости** — какие файлы нужны, какие субагенты

### Индикация tier

Каждый ответ начинается с маркера:

```
[T2] Добавляю раздел в документ...
[T4/Opus] Проектирую архитектуру компонента...
```

### Правило эскалации

1. Начинай с **нижнего** tier
2. Если результат недостаточен — эскалируй на +1
3. Никогда не начинай с T5

## Экономия по сценариям

| Сценарий | Без протокола | С протоколом | Экономия |
|----------|---------------|--------------|----------|
| «Найди файл X» | Opus + thinking | Sonnet, no think | ~90% |
| «Добавь раздел» | Opus + thinking + субагент | Sonnet, no think, без субагента | ~80% |
| «Напиши документ» | Opus + high think + 2 субагента Opus | Sonnet + low think + 1 субагент Sonnet | ~60% |
| «Спроектируй архитектуру» | Opus + max think + 3 субагента Opus | Opus + high think + 2 субагента Opus | ~30% |

**Средняя экономия при типичном распределении задач (60% T1-T2, 30% T3, 10% T4-T5): ~50-70%.**
